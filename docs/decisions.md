# Decision Log (Журнал решений)

Этот документ фиксирует ключевые решения, принятые в проекте Telegram-бота для интеграции с МИС.

---

## 1. MVP: Без истории заявок и сохранения черновиков

**Решение:** Не реализовывать историю заявок и сохранение черновиков в MVP.

**Обоснование:**
- Фокус на основной ценности: автоматизация передачи данных
- Сокращение времени разработки на 30-40%
- Отсутствие необходимости в БД на первом этапе
- Проще тестирование и отладка

**Компромисс:** При ошибке API менеджер начинает ввод заново.

**Когда пересмотрим:** После получения обратной связи от первых пользователей.

---

## 2. Валидация телефона на стороне бота

**Решение:** Нормализация и валидация номера телефона происходит в боте до отправки в МИС.

**Обоснование:**
- Раннее обнаружение ошибок (Fail Fast)
- Снижение нагрузки на МИС
- Мгновенная обратная связь для менеджера
- Единый формат данных: `7XXXXXXXXXX`

**Альтернатива:** Валидация в МИС (создает лишние запросы, приводит к необходимости начинать сначала).

---

## 3. Редактирование любого поля

**Решение:** Менеджер может отредактировать любое поле на шаге предпросмотра, не начиная сначала.

**Обоснование:**
- UX важнее сложности реализации
- Экономия времени менеджера (не нужно заново заполнять корректные поля)
- Снижение риска ошибок при повторном вводе

**Реализация:** Кнопка "Редактировать" → выбор поля → возврат к шагу ввода → предпросмотр.

---

## 4. Минимальная длина комментария: 5 символов

**Решение:** Комментарий должен содержать минимум 5 символов.

**Обоснование:**
- Баланс между контролем качества и удобством
- Исключение пустых/бессмысленных комментариев ("ок", "123")
- Достаточно для краткого описания проблемы

**Альтернативы:**
- Без минимума (риск пустых комментариев)
- 10+ символов (избыточно)

---

## 5. Статичный список филиалов

**Решение:** Список филиалов зашит в код бота, обновляется 1-2 раза в год.

**Обоснование:**
- Простота реализации
- Редкие изменения бизнес-структуры
- Не требует админ-панели
- Ограничение со стороны API МИС

**Компромисс:** Для обновления нужен деплой.

**Когда пересмотрим:** Если филиалы начнут меняться чаще 1 раза в квартал.

---

## 6. Каналы привлечения: внешняя конфигурация

**Решение:** Список каналов вынесен в конфиг для частого обновления без изменения кода.

**Обоснование:**
- Маркетинговые каналы меняются часто (кампании, акции)
- Не требуется участие разработчика
- Гибкость для бизнеса

**Отличие от филиалов:** Частота изменений.

**Выбор способа обновления каналов оставлен за исполнителем**

---

## 7. Синхронный API-запрос в МИС

**Решение:** Бот отправляет POST-запрос и ждет ответ (200/5XX) перед показом статуса.

**Обоснование:**
- Простая архитектура
- Гарантированная доставка статуса
- Легче отладка

**Альтернатива:** Асинхронный паттерн (избыточна для MVP).

---

## 8. Обработка ошибки 5XX: сброс данных

**Решение:** При ошибке сервера бот показывает сообщение и возвращает в главное меню без сохранения данных.

**Обоснование:**
- Простота реализации
- Избегание неконсистентных состояний
- Менеджер может повторить попытку

**Компромисс:** Потеря введенных данных.

**Когда пересмотрим:** Если ошибки API станут частыми (>5% запросов).

---

## 9. Telegram как канал связи

**Решение:** Бот реализуется в Telegram.

**Обоснование:**
- Уже используется менеджерами
- Не требует установки дополнительного ПО
- Быстрая разработка (готовые библиотеки)
- Поддержка кнопок и inline-запросов

---

## 10. BPMN: Разделение пулов по системам

**Решение:** Каждый участник (Клиент, Менеджер, Бот, МИС, Колл-центр) — отдельный пул.

**Обоснование:**
- Четкое разделение ответственности
- Наглядность взаимодействия систем
- Соответствие стандарту BPMN 2.0
- Упрощает передачу ТЗ разработчикам

---

- Перед изменением функционала — проверь, не нарушает ли это принятые решения
